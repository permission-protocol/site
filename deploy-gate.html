Deploy Gate

Enforce receipt redemption before production execution. No receipt → no deploy.

How It Works

Deploy Gate inserts a verification step into your CI pipeline. Before production deployment can proceed, the system must present a valid authorization receipt from Permission Protocol.

PR Merged to Main
↓
CI Build & Test
↓
Permission Protocol Verify(receipt required)
↓
Production Deploy

If verification fails or no receipt exists, the pipeline halts. This is fail-closed enforcement — the default state is denial.

GitHub Action Integration

Add the verification step to your deployment workflow:

- name: Verify Deploy Authorization
  uses: permission-protocol/verify-action@v1
  with:
    api-key: ${{ secrets.PP_API_KEY }}
    scope: |
      {
        "scopeType": "GIT_REF_COMMIT",
        "repo": "${{ github.repository }}",
        "ref": "${{ github.ref }}",
        "commitSha": "${{ github.sha }}",
        "env": "production",
        "capability": "deploy:production"
      }

- name: Deploy to Production
  if: success()
  run: ./deploy.sh production

Receipt Verification

The verify action checks:

  • Receipt exists — an authorization was explicitly issued
  • Signature valid — receipt was signed by the authority domain
  • Scope matches — receipt covers this exact commit, repo, and environment
  • Not expired — receipt is within validity window
  • Not revoked — receipt has not been cancelled

Fail-Closed Behavior

Deploy Gate defaults to denial. A deploy proceeds only when all conditions are satisfied:

$ pp verify --scope '{"scopeType":"GIT_REF_COMMIT",...}'
 
Scope: GIT_REF_COMMIT
Repo: org/service
Commit: a81f2c3e...
Environment: production
 
Receipt: VERIFIED
Signer: production-authority
Expires: 2025-02-20T00:00:00Z

Environment-Scoped Authority

Authority is scoped to environments. A staging receipt cannot authorize a production deploy. Each environment has its own signer domain with independent approval chains.

Scoping

env: staging — approved by staging-authority

env: production — requires production-authority

Integration Requirements

  • GitHub repository with Actions enabled
  • Permission Protocol API key with receipts.verify capability
  • Deploy workflow modified to include verification step
  • Approval flow configured in Permission Protocol dashboard
