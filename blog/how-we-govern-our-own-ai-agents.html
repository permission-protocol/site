Field Report AI Governance

How We Govern Our Own AI Agents

March 4, 2026  ·  Rod Chalski & Charles

On March 2nd, we sat down and asked a simple question: what is actually running on our infrastructure right now? We expected to count maybe five or six cron jobs. We found seventeen autonomous processes. That's when we built a governance system — not as a product demo, but as a survival requirement.

This is a field report, not a pitch. The code is real. The incidents are real. Here's what we did, what we found, and why governance of AI agents is harder than it looks.

01We Didn't Know What Was Running

We run Charles — an autonomous agent that handles our product board, writes code, creates PRs, sends outreach drafts, runs daily planning cycles, and monitors our own operations. Charles spawns cron jobs. Those cron jobs spawn sub-agents. Sub-agents create board items that eventually get executed. By another cron. That we set up six weeks ago.

There's no "ps aux" for this. There's no unified registry. Each job was set up when it made sense, with whatever scope felt appropriate at the time. The result was a production system running on vibes.

The question that broke us: "Which of these processes can post publicly without a human seeing it first?"

We couldn't answer it. That's a security problem.

02The Inventory: 17 Processes, 4 Authority Levels

We audited every cron job, background agent, and scheduled task. The full inventory went into an Autonomous Process Ledger — a Markdown file with a strict rule: no process runs without a ledger entry and a review date.

We assigned each process an authority level:

Level Description Examples
A Read-only — no writes anywhere Metrics collection, incident scanning
B Internal write — repos and databases only, no public actions PR drafts, board moves, memory updates
C External/public action — can affect the outside world Published blog posts, merged PRs, deployed infra, LinkedIn posts
D Prohibited — must not exist autonomously Payments, access control changes, secret rotation

The final count: 7 Authority A, 5 Authority B, 2 Authority C, 3 infrastructure services. The two Authority C processes were the ones that could take irreversible public action. Neither had a real gate.

03What We Found: Authority C With No Gate

PP-001 is our product board executor — the cron that runs five times a day, Mon–Sat, picking items from the Ready column and executing them. It can write code, create PRs, move board items, and post content. Authority C. Human gate: "Rod reviews Backlog → Ready manually."

That gate is real. But it's a workflow gate, not an enforcement gate. The agent could, in principle, create a PR without Rod's knowledge if something in the flow was misconfigured. There was no cryptographic proof that the action was reviewed.

PP-008 (LinkedIn weekly poster) had the same problem. Authority C. Human gate: "Telegram approval." But the old approve.sh script accepted a content hash as a parameter — meaning the agent could technically pass its own hash. Prompt-level instructions said "don't do this." That's not enforcement. That's a prayer.

The pattern we kept finding: "Authority C" processes with gates defined as human intention, not machine-enforced checkpoints. The agent knew it needed approval. It just had no mechanism that actually required it.

04What We Enforced: Tokens, Drift Detection, SLA Auto-Disable

Three changes made the system actually enforceable:

1. Cryptographic approval tokens

We replaced the old approve.sh with mint-approval-token.sh. The new script enforces three properties:

  • Telegram sender ID must match Rod's hardcoded user ID — not agent-configurable
  • Content hash is computed from the actual draft file — never accepted as a parameter
  • UUID nonce per mint — prevents replay attacks

The gate script (pp-001-pr-gate.sh) is called at execution boundary — right before git push or PR creation — not at intent stage. If no valid token exists, the action is blocked and the item moves to "Blocked (Human Input Required)" with the draft hash. Fail-closed by design.

Here's the core of the verification loop — the part that actually enforces the boundary:

#!/bin/bash
# pp-001-pr-gate.sh — called at git push / PR create boundary
# Exit 0 = approved. Exit 1 = denied (PR blocked).

DRAFT_FILE="$1"
APPROVALS_DIR="$WORKSPACE/company-os/governance/approvals"

# Compute hash of the actual draft
DRAFT_HASH=$(sha256sum "$DRAFT_FILE" | awk '{print $1}')

# Walk approval tokens
APPROVED=false
for token_file in "$APPROVALS_DIR"/*.json; do
  [ -f "$token_file" ] || continue

  python3 - < exp:
    sys.exit(1)

# 4. Sender must be Rod (double-check — minting script also enforces this)
if t.get("sender_id") != "6091356144":
    sys.exit(1)

sys.exit(0)
PYEOF

  if [ $? -eq 0 ]; then
    APPROVED=true
    break
  fi
done

if [ "$APPROVED" = "true" ]; then
  echo "✅ PR gate: approved (token verified)"
  exit 0
else
  echo "🔴 PR gate: denied — no valid approval token for this draft"
  exit 1
fi

2. Drift detection

Every morning at 7 AM, a governance check scans the live cron state and compares it against the ledger. If a new process appears that isn't registered — or if a process's schedule has changed without a ledger update — it flags a HIGH severity finding and writes it to drift-log.jsonl.

The day after we deployed this, it caught two unregistered processes: governance-integrity-check (which we'd just created) and skills-auto-update (which had been running silently). Both were real. Neither was in the ledger yet.

3. Review SLA auto-disable

Every Authority B and C process must be reviewed every 30 days. If last_reviewed is > 30 days old, the governance check auto-disables the process and logs a breach. Re-enabling requires updating the ledger, not just flipping a flag. Warning at 21 days. No exceptions.

05What Surprised Us: Governance Must Govern Itself

Here's the recursive problem nobody warns you about.

GOV-001 is the cron that runs the governance integrity check. It's Authority A — read-only, no external actions, no human gate. That all makes sense. But it logs its findings to drift-log.jsonl. And the thing that monitors drift-log.jsonl for freshness is... also GOV-001.

The agent is checking itself.

If GOV-001 silently fails — crash, timeout, OpenClaw outage — the drift log goes stale. Nothing alerts. Your Authority C processes are running ungated, and the system designed to catch that has quietly stopped working. You won't know until something breaks publicly.

The principle we named: Any governance system that depends on the system it governs to report its own failures is not governance — it's a diary.

The fix is issue #257 on our board right now: a dead man's switch that alerts if drift-log.jsonl has no entry for 48 hours, triggered by an external mechanism that doesn't flow through Charles at all. GitHub Actions, external uptime monitor, or a Telegram webhook on a separate server. Independent by design.

06Architecture: The Full Approval Flow

── Authority C Action Flow ────────────────────────────────────────── Agent (PP-001 cron, 5x daily) │ ├─ picks Ready item from board │ └─ Rod is the only path from Backlog → Ready │ ├─ drafts content / code changes │ └─ writes to /tmp, does NOT push yet │ ├─ GATE CHECK: pp-001-pr-gate.sh --draft-file <path> │ ├─ looks for approval token in governance/approvals/ │ ├─ verifies: process_id, content_hash, expiry, sender_id │ └─ if no valid token → BLOCKED │ ├─ item → "Blocked (Human Input Required)" │ └─ notifies Rod with draft hash via Telegram │ ├─ [token valid] → git push + create PR │ └─ PR includes Review Guide comment (mandatory) │ └─ item → QA Review (not Done — QA pipeline verifies first) ── Token Minting Flow (Rod's side) ────────────────────────────────── Rod (Telegram) │ ├─ receives draft + hash notification ├─ reviews content └─ sends approval signal to Telegram bot └─ mint-approval-token.sh ├─ verifies sender ID = 6091356144 (hardcoded) ├─ reads draft file, computes hash itself ├─ generates UUID nonce (no replay) ├─ writes token to governance/approvals/<uuid>.json └─ appends to approval-log.jsonl (append-only)

07What This Cost Us (And What It Bought)

The governance build took about four hours. Audit, ledger, scripts, tests. The recurring cost is roughly 10 minutes per new process (ledger entry + Rod review) and a brief daily check. That's it.

What we got: a system where every autonomous action has a traceable authority level, every external action requires a cryptographic token Rod explicitly minted for that exact content, and every deviation from the ledger is caught within 24 hours.

We also got a dashboard. Not because we needed another dashboard, but because the audit trail was already there — we just rendered it. Current state of all 17 processes, authority distribution, SLA countdowns, last drift detection result.

We're not special. The reason we had 17 undocumented processes is the same reason everyone else does: you build them one at a time, when they make sense, and you never stop to ask what the whole looks like. The governance system didn't require new infrastructure. It just required sitting down and being honest about what was running.

The rule we now enforce: No autonomous process without a ledger entry, an authority level, and a Last Reviewed date. If you can't answer "what happens if this runs without a human knowing?" in one sentence, it doesn't run yet.
