Receipt Model

Cryptographic authorization artifacts bound to specific actions.

Receipt Payload Structure

A receipt is a signed JSON document containing the authorization decision and the scope it covers. The payload includes all fields necessary to verify the authorization at execution time.

{
  "id": "rcpt_7f3d8a2e...",
  "companyId": "comp_abc123",
  "version": "1.0",
  "status": "APPROVED",
  
  "scope": {
    "scopeType": "GIT_REF_COMMIT",
    "repo": "org/service",
    "ref": "refs/heads/main",
    "commitSha": "a81f2c3e9b4d...",
    "env": "production",
    "capability": "deploy:production"
  },
  
  "agentId": "agent_deploy_bot",
  "requestId": "req_5c2a...",
  "policyVersion": "v2024.1",
  "riskTier": "TIER_1",
  
  "createdAt": "2025-02-19T10:30:00Z",
  "expiresAt": "2025-02-19T12:30:00Z",
  
  "signature": {
    "algorithm": "Ed25519",
    "keyId": "key_prod_001",
    "value": "base64_encoded_signature..."
  }
}

Scope Binding

The scope defines exactly what action this receipt authorizes. Scope binding ensures a receipt cannot be reused for a different action.

GIT_REF_COMMIT Scope

For production deploys tied to a specific commit:

{
  "scopeType": "GIT_REF_COMMIT",
  "repo": "org/service",
  "ref": "refs/heads/main",
  "commitSha": "a81f2c3e9b4d5f6a7b8c9d0e1f2a3b4c5d6e7f8a",
  "env": "production",
  "capability": "deploy:production"
}

PR_HEAD Scope

For PR-level checks before merge:

{
  "scopeType": "PR_HEAD",
  "repo": "org/service",
  "prNumber": 142,
  "headSha": "b92c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c",
  "env": "production",
  "capability": "deploy:production"
}

A receipt for commitSha: a81f2c3 cannot authorize a deploy of commitSha: b92c4d5. Scope mismatch results in verification failure.

Signature Scheme

Receipts are signed using Ed25519. The signature covers a canonicalized representation of the receipt payload, ensuring any modification invalidates the signature.

Canonicalization

Before signing, the payload is canonicalized:

  • Fields sorted alphabetically
  • No whitespace in JSON encoding
  • Consistent date format (ISO 8601)
  • Null values excluded

Signed Fields

The signature covers these fields:

id, companyId, idemKey, agentId, runId, requestJson,
inputHash, status, riskTier, policyVersion, reasonCodes,
summary, receiptVersion, canonicalization, signatureAlg,
signatureKeyId, expiresAt, createdAt

Modifying any signed field — including expiresAt — invalidates the signature.

Key Identification

Each signature includes a keyId identifying which signing key was used. Verifiers resolve the key from Permission Protocol's key registry. Keys are rotated periodically; old keys remain valid for verification of existing receipts.

Replay Protection

Receipts include multiple replay protection mechanisms:

Expiration

Every receipt has an expiresAt timestamp. After expiration, the receipt cannot authorize any action, even if the signature remains valid.

Single-Use Redemption

When a receipt is used to authorize an action, it is marked as redeemed in the ledger. A redeemed receipt cannot be used again.

Scope Uniqueness

The scope hash ensures only one valid receipt exists for a given scope at any time. Creating a new receipt for the same scope invalidates the previous receipt.

Verification Flow

Verification is a single API call that returns a boolean decision:

POST /v1/receipts/verify
Authorization: Bearer pp_live_xxx

{
  "scope": {
    "scopeType": "GIT_REF_COMMIT",
    "repo": "org/service",
    "ref": "refs/heads/main",
    "commitSha": "a81f2c3e...",
    "env": "production",
    "capability": "deploy:production"
  }
}

Response on success:

{
  "verified": true,
  "receiptId": "rcpt_7f3d8a2e...",
  "expiresAt": "2025-02-19T12:30:00Z"
}

Response on failure:

{
  "verified": false,
  "errorCode": "RECEIPT_NOT_FOUND"
}

Error Codes

  • RECEIPT_NOT_FOUND — no receipt exists for this scope
  • RECEIPT_EXPIRED — receipt exists but has expired
  • RECEIPT_REVOKED — receipt was explicitly revoked
  • SIGNATURE_INVALID — receipt signature verification failed
  • SCOPE_MISMATCH — receipt scope does not match request

Idempotency

Authorization requests include an idempotency key (idemKey). Submitting the same request twice returns the existing receipt rather than creating a duplicate. This allows safe retry of failed requests without risk of duplicate authorizations.

{
  "idemKey": "deploy-org/service-a81f2c3e-production",
  "scope": { ... }
}

Audit Trail

Every receipt creation, verification attempt, and redemption is logged in the receipt ledger. The audit trail includes:

  • Request metadata (agent, timestamp, source IP)
  • Policy evaluation results
  • Approval chain (who approved, when)
  • Verification attempts (success/failure)
  • Redemption event

The ledger is append-only. Records cannot be modified or deleted.
