Request Access

Architecture

Trust boundary separation between execution systems and authority domains.

System Components

Agent

The autonomous system requesting to perform an action. An agent may have broad platform access but cannot self-authorize irreversible operations. Agents initiate requests; they do not issue receipts.

Execution Platform

The system where the action will occur — GitHub, CI runner, cloud provider, payment processor. The execution platform enforces receipt verification before action execution.

Permission Router

Routes authorization requests to the appropriate policy engine based on action type, environment, and risk tier. Handles request intake and response delivery.

Policy Engine

Evaluates requests against configured policies. Determines approval requirements — automatic approval for low-risk actions, human review for high-risk operations, multi-party approval for critical changes.

Receipt Ledger

Immutable record of all authorization decisions. Every approval generates a signed receipt stored in the ledger. Receipts can be verified against the ledger at any time.

Signer Domain

The cryptographic authority that signs receipts. Each environment (staging, production) has its own signer domain. Signer domains are isolated — a staging signer cannot produce a production receipt.

Enforcement Layer

The verification step in your execution pipeline. Before an irreversible action proceeds, the enforcement layer validates that a matching receipt exists and is valid.

The execution domain can request authorization. It cannot grant it. The authority domain issues receipts but does not execute actions. Neither domain can impersonate the other.

Authority Issuance vs Execution

Traditional systems conflate authorization with execution. If you can trigger a deploy, you can deploy. Permission Protocol separates these concerns:

  • Execution capability — the ability to run deploy.sh
  • Authorization receipt — the cryptographic proof that this deploy was approved

An agent may have execution capability without authorization. The execution platform will not proceed without both.

Receipt Lifecycle

  1. Request — Agent submits authorization request with scope
  2. Evaluation — Policy engine determines approval requirements
  3. Approval — Human or automatic approval based on policy
  4. Signing — Signer domain produces cryptographic receipt
  5. Storage — Receipt recorded in ledger
  6. Verification — Enforcement layer validates receipt at execution time
  7. Redemption — Action executes; receipt marked as redeemed

Receipts are single-use. Once redeemed, they cannot authorize another action.

Verification Model

Receipt verification is stateless from the execution platform's perspective. The enforcement layer sends the scope to Permission Protocol and receives a yes/no decision. The execution platform does not need to understand policy logic — only whether a valid receipt exists.
